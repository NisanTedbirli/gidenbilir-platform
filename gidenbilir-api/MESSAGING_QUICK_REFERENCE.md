## Database Schema

### Conversations Table
```
Id (PK)
User1Id (FK ‚Üí Users) ‚Äî Lower of two user IDs (normalized)
User2Id (FK ‚Üí Users) ‚Äî Higher of two user IDs (normalized)
CreatedAt (UTC)
LastMessageAt (UTC) ‚Äî Updated on new message
```

### Messages Table
```
Id (PK)
Content (string, 1-2000 chars)
SenderId (FK ‚Üí Users)
ConversationId (FK ‚Üí Conversations, CASCADE)
CreatedAt (UTC)
```

### Indexes
- `IX_Conversations_User1_User2` (UNIQUE) ‚Äî Prevent duplicate conversations
- `IX_Conversations_LastMessageAt` ‚Äî Sort conversations by latest activity
- `IX_Messages_ConversationId` ‚Äî Fast message lookup
- `IX_Messages_CreatedAt` ‚Äî Pagination support

## Testing

### Quick Test Flow
```bash
# 1. Register two users (via /api/auth/register)
# 2. Create conversation
POST /api/conversations
{ "participantId": 2 }

# 3. Send message
POST /api/conversations/1/messages
{ "content": "Merhaba!" }

# 4. Get messages (paginated)
GET /api/conversations/1/messages?page=1&pageSize=10

# 5. List conversations
GET /api/conversations

# 6. Delete message
DELETE /api/conversations/1/messages/1
```

### Validation Rules
- `ParticipantId` ‚Äî Valid user ID (not self)
- `Content` ‚Äî 1-2000 characters, non-null
- `Page` ‚Äî Min 1, default 1
- `PageSize` ‚Äî Min 1, default 10, max reasonable limit
- JWT token ‚Äî Valid Bearer token required for all endpoints

## Security Features

‚úÖ JWT Authentication ‚Äî All endpoints protected  
‚úÖ User Isolation ‚Äî Can only see own conversations/messages  
‚úÖ Sender-Only Delete ‚Äî Only message sender can delete  
‚úÖ Conversation Ownership ‚Äî Both users must be participants  
‚úÖ Unique Conversation Pairs ‚Äî No duplicate conversations with same user
